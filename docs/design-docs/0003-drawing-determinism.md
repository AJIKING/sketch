# ADR 0003: 描画の決定性(乱数 seed と時間源の注入)

- 状態: 採用
- 日付: 2026-06-14

## 決定

描画エンジンの非決定要素を、注入した境界の裏に閉じ込めて**決定的にテストできる**ようにする。

- ブラシの散らし(scatter)・粒状感(jitter)に使う乱数は、**`Random(seed)` を注入**する。グローバルな `Random()` / `math.Random.secure()` を直接呼ばない。
- ストロークの速度(→ ink の筆幅)・保存時刻に使う時間は、**`Clock` を注入**する。`DateTime.now()` / `Stopwatch` 直叩きをしない。
- ストローク → スタンプ列(位置・半径・α・色)の計算は **pure Dart の domain 層**(`stroke_planner`)に置き、`dart:ui` でのラスタライズと分離する。

## 背景

プロトタイプ(`docs/prototype/hatch-sketch-app.html`)は `Math.random()` と `performance.now()` を描画ループ内で直接呼んでいる。このまま移植すると:

- 同じ入力でも描画結果が毎回変わり、golden / unit test で検証できない。
- 失敗を再現できない(seed が残らない)。
- 速度依存の筆致が実時間に縛られ、テストが遅く flaky になる。

候補:

| 候補 | 評価 |
| --- | --- |
| グローバル乱数・実時間をそのまま使う | 実装は早いが検証不能・flaky。ハーネス方針に反する |
| 乱数・時間を注入し、計算を domain に分離 | 描画結果を固定 seed で再現でき、unit / golden で守れる |

## 理由

- `docs/harness-engineering.md` の原則「テストは実時間・外部状態に依存しない」「flaky はハーネスの欠陥」に直結する。
- scatter を含むストローク計画を固定 seed で再現できれば、ブラシのリグレッションを安い unit test と限定的な golden で捕まえられる。
- 計算(domain)とラスタライズ(ui)を分けることで、重い画素処理に触れずにロジックを検証できる。

## 結果

- `stroke_planner` は `(前点, 現点, 速度, BrushPreset, Random)` を受け取り、スタンプ列を返す純関数とする。同じ引数なら同じ出力。
- 速度は `Clock` から得た時刻差で算出する。テストでは fake clock を手動で進める。
- 失敗時のログに seed を含める(再現用)。
- ラスタライズ結果(画素)の golden は、固定 seed・固定ストロークで生成したものに限る。自由描画は対象外(ADR 0002)。

## 既知の限界: undo 履歴が保持する `ui.Image` の解放

undo/redo のスナップショット(`History<UndoEntry>`、ADR 0004 のラスター画像)は `ui.Image` を**参照で**保持する。`PixelEdit` は変更前の 1 枚、`StackEdit`(結合・マスク等の構成変更)は全レイヤー本体 + 全マスクを保持する。現状、履歴上限(既定 16)を超えて押し出されたエントリや、新規記録で破棄される redo エントリの `ui.Image` を **明示 `dispose` していない**(GC ファイナライザ任せ)。高解像度ドキュメントでは累積メモリが大きくなりうる。

安易に「破棄するエントリの画像を dispose する」と壊れる: スナップショットは画像オブジェクトを**共有**しうる(例: 構成スナップショットは未変更レイヤーの**ライブ画像**をそのまま参照、複数の構成スナップショットが同一画像を共有)。そのため、ライブ参照・他履歴エントリとの**参照カウント**を持たずに dispose すると use-after-dispose で落ちる。正しい解放にはスナップショット画像の参照カウント機構が必要で、別タスクとする。当面の緩和として、レイヤー削除・結合で**不要になったライブ参照は外す**(`set(id, null)`。dispose はしない=履歴から復元可能)。
