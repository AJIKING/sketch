# ADR 0004: ラスター(ピクセル)キャンバスエンジンへの移行

- 状態: 採用
- 日付: 2026-06-14

## 決定

各レイヤーを**ピクセルバッファ(`dart:ui.Image`)**として保持するラスターエンジンへ移行する。立ち上げ時のベクター方式(`PaintedStroke` の点列再生)は、ライブ描画の一時表現としてのみ残し、ストローク確定時にレイヤー画像へ焼き込む(bake)。

- **レイヤー実体**: `ui.Image?`(null = 透明)。
- **ライブ描画**: 描画中のストロークはベクターで上に重ねて表示し、ポインタ離上時に `PictureRecorder` で「既存画像 + ストローク」を 1 枚へ焼き込む。
- **undo/redo**: スナップショットはレイヤー画像の参照(`ui.Image` は不変なので保持は安価)。`CanvasSurface` 境界はそのまま使える(不透明トークン = 画像)。
- **ピクセル操作**(塗りつぶし・フィルタ): `image.toByteData()` → `Uint8List`(RGBA)で加工 → `decodeImageFromPixels` で再構成。アルゴリズムは pure Dart に置き unit テストする。
- **合成**: レイヤー画像をブレンドモード・不透明度・クリッピング・アルファロックを適用して重ねる。

## 背景

ibisPaint 系の主要機能 —— 塗りつぶし(バケツ)、ブレンドモード、ぼかし等フィルタ、アルファロック、クリッピング、水彩、スマッジ —— はいずれもレイヤーの**ピクセル**を読み書きする。ベクター方式(点列再生)はこれらを表現できない。

| 候補 | 評価 |
| --- | --- |
| ベクター継続 | 大改修は不要だが、塗りつぶし・フィルタ・アルファロック等が原理的に作れない |
| 完全ラスター(毎フレーム toImage) | 正しいが、ライブ描画で毎ストローク全面 toImage は重く janky |
| ハイブリッド(確定画像 + ライブ上書き、離上で焼込) | ライブは滑らか、確定はピクセル。ibis 系機能の土台になる |

## 理由

- ユーザー要望(ibis 相当機能の網羅)を満たすにはピクセルエンジンが必須。
- ハイブリッドにすることで、ライブ描画のなめらかさと確定後のピクセル操作を両立する。
- `CanvasSurface` 境界(snapshot/restore/clear が不透明 `Object`)を維持できるため、`CanvasController` と履歴ロジックは変更不要。影響を ui 層に閉じ込められる。
- ピクセルアルゴリズム(flood fill, フィルタ)を pure Dart の `Uint8List` 操作として書けば、`dart:ui` に触れず unit テストできる。

## 結果

- レイヤーは `ui.Image?` を持つ。空レイヤーは null(透明)。
- 焼き込みは非同期(`Picture.toImage`)。1 ストローク 1 回の async は許容する。
- 履歴は画像参照を保持。上限 16(既存)。タイル化(差分保存)は重くなったら別 ADR で検討する。
- 既存の golden(ベクターのブラシプレビュー)は焼込結果に依存しないため維持できる。
- 性能(大きいキャンバスでの toImage / toByteData コスト)が問題化したら、タイル分割・GPU テクスチャ・差分履歴を別 ADR で検討する。
- ブレンド/クリッピング/アルファロック/フィルタ等の個別仕様は後続 ADR と `docs/product-spec.md` の追補で定義する。
